# 並行チャンク分割アップロード設計

*最終更新: April 10, 2026 at 7:14 PM PDT*

## この文書の目的

この文書は、BurnBox 2.0.0 における最も重要な技術選定、すなわちチャンク分割 + multipart アップロード構成の採用理由を記録するためのものです。

これは本プロジェクトで最も難しい技術領域です。理由は、ブラウザ、ネットワーク揺らぎ、Worker 実行、R2 multipart、D1 状態整合の境界がすべてここに集中しているからです。

## 問題設定

BurnBox が扱うのは小さな文書だけではありません。

- インストーラ
- デスクトップ向けイメージ
- バイナリアーカイブ
- 大きなリリース成果物

この条件では、単発の single upload は十分に安定しませんでした。実際に観測された症状は次のとおりです。

- アップロードが終わったように見えても一覧に出ない
- 次のアップロードで前のファイルが見えるようになる
- 大きいファイルほど不安定になる
- 転送完了とメタデータ確定の違いが利用者に見えにくい

## 採用した方式

BurnBox は現在、次の構成を採用しています。

- 5 MiB チャンク
- ブラウザ側でのファイル分割
- Worker 経由のチャンク転送
- R2 multipart 組み立て
- D1 による upload plan / uploaded part の永続化

この設計を「並行チャンク分割アップロード」と呼ぶのは、チャンク単位・multipart 前提・制御された並行転送に対応可能な構造だからです。現時点の本番寄り設定では、安定性を優先して保守的な実行順序を採用しています。

## なぜ 5 MiB か

5 MiB は multipart の実運用上の基準として扱いやすく、次の点でバランスが取れています。

- 失敗時の再送コストが小さい
- リクエスト数が増えすぎない
- Worker の request body 上限から十分に余裕がある
- 大きすぎるチャンク実験より安定する

## なぜブラウザ直結の multipart presigned URL ではないのか

理論上は可能ですが、BurnBox の現在の段階では複雑さが大きすぎます。

- クライアントで presign のライフサイクルを多く管理する必要がある
- multipart 状態がブラウザ側に多く露出する
- CORS 面が広がる
- 障害切り分けが難しくなる

BurnBox は極端な最大スループットよりも、運用上の見通しと復旧容易性を優先します。そのため、Worker を通す方式が適切でした。

## 技術的な意味

この方式により、システムは次の 4 点で改善されます。

1. 単一巨大リクエストを主要な失敗境界にしない
2. クライアント申告ではなくサーバー主導の upload state を持てる
3. 転送進捗と ready 状態を分離できる
4. 信頼性を副産物ではなく設計要件として扱える

## 主な技術難所

### 1. 転送完了と ready は同じではない

ブラウザが送信を終えても、ファイルがすぐに利用可能になるとは限りません。BurnBox は finalization を明示的に扱います。

### 2. ストレージ識別子はサーバーが支配すべき

後で正式な file record になる object identity をクライアントに決めさせるべきではありません。そのため upload planning と multipart finalization はサーバー側で実行します。

### 3. 進捗表示は本当に正しくなければならない

偽の progress bar は、進捗表示がないより悪いです。BurnBox はチャンク転送進捗を実際の part 状態に結びつけ、最後の組み立ては別フェーズとして見せます。

### 4. edge upload の信頼性は UI だけでは解けない

本当の難しさは、ブラウザ分割、ネットワーク再試行、Worker 処理、R2 multipart、D1 協調の相互作用にあります。

## 実装上の難所

### Upload state machine

複数リクエストをまたいで次の状態を保持する必要があります。

- created
- uploading
- processing
- ready
- failed

### Part 管理

各 part には安定した番号と永続記録が必要です。欠落や重複は最終組み立ての不整合につながります。

### Finalization UX

高い進捗率で停止して見えると、利用者は故障と判断します。そのため BurnBox は chunk upload と finalization を明確に分けて表示します。

### リカバリ意味論

システムは次の場合でも整合的である必要があります。

- あるチャンクだけ失敗する
- ブラウザが再試行する
- 全チャンク送信後に finalization が失敗する
- 転送完了と metadata commit の成否がずれる

## 技術的価値

この設計は単なる upload 最適化ではありません。BurnBox 全体の信頼性を支える中心技術です。

- 一覧表示が信頼できる
- share 発行が ready 状態に依存できる
- 削除挙動が理解しやすくなる
- オペレーターがファイルの現在位置を把握しやすくなる

## 研究的価値

BurnBox は、edge-native な運用ツールが「楽観的な試作」から「状態を持つ failure-aware な設計」へ進化する過程を示す小さな実例です。
